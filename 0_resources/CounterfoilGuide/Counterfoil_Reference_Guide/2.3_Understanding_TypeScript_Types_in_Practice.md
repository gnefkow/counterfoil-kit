# 2.3: Understanding TypeScript - Types in Practice

## Goal

Teach you how to read and understand TypeScript type syntax as it appears in Counterfoil. By the end of this chapter, you'll be able to interpret type definitions in component files, understand what props a component accepts, and recognize the patterns the toolkit uses. (See [2.2 What Are Types and Why Do They Matter?](2.2_What_Are_Types_and_Why_Do_They_Matter.md) for the conceptual foundation.)

## What TypeScript Is: JavaScript with Type Annotations

TypeScript is JavaScript with an added layer: type annotations. An annotation is extra syntax that describes what a value is—a string, a number, an object with specific properties, a function that takes certain arguments. TypeScript is a *superset* of JavaScript: every valid JavaScript file is valid TypeScript. The difference is that TypeScript lets you (and often requires you to) add type information.

In practice, you write normal JavaScript and sprinkle in type annotations where they help. A variable might be annotated as `const name: string`. A function parameter might be annotated as `(value: string) => void`. The annotations don't change what the code does at runtime—they're used by the TypeScript compiler to check your code before it becomes JavaScript.

## How TypeScript Works: Check First, Then Compile to JavaScript

When you save a file or run a build, TypeScript does two things: it *type-checks* your code, and it *compiles* (transpiles) it to JavaScript. Type-checking means the compiler reads your annotations and validates that you're using values correctly. If you pass a number where a string is expected, or access a property that doesn't exist, the compiler reports an error. Compilation means stripping out the type annotations and converting modern syntax to JavaScript that browsers can run.

The important part: type errors block the build. If you have a type error, the compiler will fail and you won't get JavaScript output until you fix it. That's intentional—it forces you to correct mistakes before the code runs in a browser.

When you're reading Counterfoil code, the `.ts` and `.tsx` files are TypeScript. The build pipeline (Vite) runs the TypeScript compiler, which produces JavaScript in the `dist/` folder. As a reader, you're looking at the source TypeScript; the types are there to help you understand the code and to help the compiler catch errors.

## Basic Type Syntax Explained

### Primitives: `string`, `number`, `boolean`

The simplest types are the primitives: `string`, `number`, and `boolean`. They describe the basic values you use every day.

- **`string`**: Text. Any sequence of characters in quotes. Examples: `"primary"`, `"Save"`, `""`.
- **`number`**: Numeric values. Integers and decimals. Examples: `42`, `3.14`, `0`.
- **`boolean`**: True or false. Examples: `true`, `false`.

In Counterfoil, you'll see these in prop types. For example, from `src/components/primitives/Button.tsx`:

```ts
type ButtonProps = {
  disabled?: boolean
  "aria-label"?: string
  // ...
}
```

Read `disabled?: boolean` as: *"disabled is optional, and when present, it's a boolean."* Read `"aria-label"?: string` as: *"aria-label is optional, and when present, it's a string."*

When a component receives a boolean, it's used for conditional logic (e.g., whether to show a disabled state). When it receives a string, it's usually displayed as text or used as an attribute. The type tells you what kind of value to pass.

### Arrays: `string[]` vs `Array<string>` (They're the Same)

An array is a list of values. In TypeScript, you specify what kind of items the array contains. The syntax `string[]` means *"array of strings."* The syntax `Array<string>` means the same thing—just a different style. Counterfoil typically uses the bracket form: `string[]`.

Examples:

- `string[]` — array of strings, e.g. `["a", "b", "c"]`
- `number[]` — array of numbers, e.g. `[1, 2, 3]`
- `boolean[]` — array of booleans (rare in practice)

Arrays of objects are common. From `src/components/navigation/TabBar.tsx`:

```ts
type TabBarProps = {
  tabs: {
    id: string
    label: string
    disabled?: boolean
    icon?: React.ReactNode
    iconPosition?: "left" | "right"
  }[]
  selectedId: string
  onSelect: (id: string) => void
  // ...
}
```

Read `tabs: { ... }[]` as: *"tabs is an array of objects, where each object has id (string), label (string), optional disabled (boolean), optional icon, and optional iconPosition."*

The `[]` at the end applies to the whole object type. So you have an array, and each element is an object with that shape.

### Objects: `{ name: string, age: number }`

Object types describe the shape of an object: what properties it has and what type each property is. The syntax is `{ propertyName: type, ... }`.

Example:

```ts
{ name: string; age: number }
```

Read this as: *"object with name (string) and age (number)."* Semicolons or commas between properties are both valid; Counterfoil often uses semicolons for clarity.

From `src/components/form/InputField.tsx`:

```ts
type InputFieldProps = {
  label: string
  description?: string
  helpText?: string
  errorText?: string
  type?: "text" | "email" | "password" | "number" | "search"
  value?: string
  disabled?: boolean
  onChange?: (value: string) => void
  // ...
}
```

Each line is a property. `label: string` means *"label is required and must be a string."* `description?: string` means *"description is optional, and if present, it's a string."* The `?` marks optional properties—we'll cover that next.

When you pass props to a component, you're passing an object. The type definition is the contract: it tells you exactly what properties the component expects and what types they must be.

### Union Types: `"primary" | "secondary"`

A union type means *"one of these values."* The pipe character `|` separates the options. So `"primary" | "secondary"` means *"either the string 'primary' OR the string 'secondary'."*

This is one of the most useful patterns in Counterfoil. Buttons have variants; layout components have gap sizes; text has hierarchy levels. Instead of accepting any string (which would allow typos like `"prmary"`), the type restricts the value to a fixed set.

From `src/components/primitives/Button.tsx`:

```ts
type ButtonVariant =
  | "primary"
  | "secondary"
  | "tertiary"
  | "quaternary"
  | "link"
  | "destructive-primary"
  | "destructive-secondary"

type ButtonSize = "sm" | "md" | "lg" | "2xl"
```

Read `ButtonVariant` as: *"one of these seven string literals."* Read `ButtonSize` as: *"one of 'sm', 'md', 'lg', or '2xl'."*

When you write `<Button variant="primary" />`, TypeScript checks that `"primary"` is in the union. If you write `<Button variant="prmary" />`, you get an error—and your editor will offer autocomplete with the valid options.

Union types can mix other types too, not just strings. For example, `string | number` means *"either a string or a number."* But in Counterfoil components, you'll mostly see unions of string literals like `"sm" | "md" | "lg"`.

### Optional Properties: `name?: string`

The `?` after a property name means the property is optional. The component will work whether you provide it or not.

- `label: string` — required. You must pass `label`.
- `description?: string` — optional. You can omit it, or pass a string.

From `src/components/primitives/Stack.tsx`:

```ts
type StackProps = {
  children: React.ReactNode
  gap?: StackGap
  className?: string
}
```

Here, `children` is required (no `?`). Every `Stack` must have children. But `gap` and `className` are optional. If you don't pass `gap`, the component will use a default (in this case, `"m"`). If you don't pass `className`, it simply won't add extra classes.

Read `gap?: StackGap` as: *"gap is optional, and if present, it must be one of the StackGap values."*

Optional props often have default values in the component. For example, `Stack` uses `gap = "m"` in its destructuring, so when `gap` is omitted, it behaves as if you passed `"m"`.

### Functions: `(value: string) => void`

Function types describe what a function accepts and what it returns. The syntax is `(param1: type1, param2: type2) => returnType`.

- `() => void` — function that takes no arguments and returns nothing. Common for click handlers.
- `(value: string) => void` — function that takes one string and returns nothing. Common for `onChange` handlers that pass the new value.
- `(id: string) => void` — function that takes a string (e.g. an id) and returns nothing. Common for `onSelect` handlers.

From `src/components/form/Input.tsx`:

```ts
type InputProps = {
  onChange?: (value: string) => void
}
```

Read this as: *"onChange is an optional function that takes a string (the new input value) and returns nothing."*

From `src/components/navigation/TabBar.tsx`:

```ts
type TabBarProps = {
  onSelect: (id: string) => void
  // ...
}
```

Read this as: *"onSelect is a required function that takes a string (the tab id) and returns nothing."*

Why does `Input` pass a string to `onChange` instead of an event? The component abstracts that detail. The raw DOM `input` fires `onChange` with an event; the `Input` component extracts `e.target.value` and calls your callback with just the string. So from the consumer's perspective, `onChange` receives the new value directly. The type reflects that contract.

## Type Aliases: `type ButtonVariant = "primary" | "secondary"`

A type alias is a name you give to a type so you can reuse it. Instead of writing `"primary" | "secondary" | "tertiary"` everywhere, you define it once and refer to it by name.

From `src/components/primitives/Button.tsx`:

```ts
type ButtonVariant =
  | "primary"
  | "secondary"
  | "tertiary"
  | "quaternary"
  | "link"
  | "destructive-primary"
  | "destructive-secondary"

type ButtonSize = "sm" | "md" | "lg" | "2xl"

type ButtonWidth = "hug" | "fill"

type ButtonProps = {
  variant?: ButtonVariant
  size?: ButtonSize
  width?: ButtonWidth
  disabled?: boolean
  onClick?: () => void
  children?: React.ReactNode
  // ...
}
```

`ButtonVariant`, `ButtonSize`, and `ButtonWidth` are aliases. They exist for reusability and clarity. When you add a new variant, you change it in one place; every use of `ButtonVariant` automatically gets the update. And when you read `variant?: ButtonVariant`, you know exactly what it means without inlining the full union.

Type aliases also appear in class maps. In `Button.tsx`:

```ts
const variantClasses: Record<ButtonVariant, string> = {
  primary: "bg-[var(--button-primary-bg)] ...",
  secondary: "bg-[var(--button-secondary-bg)] ...",
  tertiary: "...",
  quaternary: "...",
  link: "...",
  "destructive-primary": "...",
  "destructive-secondary": "...",
}
```

`Record<ButtonVariant, string>` means: *"an object whose keys are ButtonVariant values and whose values are strings."* So you must have an entry for every variant—no missing keys. The type checker enforces that. If you add a new variant to `ButtonVariant` and forget to add it to `variantClasses`, TypeScript will error. That's the type system helping you keep things in sync.

## Reading Type Definitions: Step-by-Step Breakdown

When you open a Counterfoil component, the type definitions are usually at the top, in a "Public API" section. Here's how to read them systematically.

Take `ButtonProps` from `src/components/primitives/Button.tsx`:

```ts
type ButtonProps = {
  children?: React.ReactNode
  variant?: ButtonVariant
  size?: ButtonSize
  width?: ButtonWidth
  disabled?: boolean
  onClick?: () => void
  icon?: React.ReactNode
  iconPosition?: "left" | "right"
  leftIcon?: React.ReactNode
  rightIcon?: React.ReactNode
  "aria-label"?: string
}
```

**Step 1: Identify required vs optional.** No `?` means required. Here, every prop has `?`, so all are optional. That means you can render `<Button>Click</Button>` with no other props and it will work (using defaults).

**Step 2: Read each property's type.** For `variant?: ButtonVariant`, you need to know what `ButtonVariant` is. Look above for the alias: it's the union of the seven variant strings. So `variant` accepts one of those.

**Step 3: Read function types.** `onClick?: () => void` means an optional function with no parameters and no return value. You'd pass something like `onClick={() => handleSave()}`.

**Step 4: Note React-specific types.** `React.ReactNode` means "anything React can render"—text, elements, arrays of elements, null. So `children` and `icon` can be flexible.

**Step 5: Check for special keys.** `"aria-label"` is in quotes because it contains a hyphen. In JavaScript, property names with special characters need quotes. The type still works the same: optional string.

From that definition, you can answer: *What props does Button accept? What's required? What are the valid values for variant and size?* You don't need to read the component implementation—the types are the interface.

## Type Inference: When TypeScript Figures It Out for You

You don't always have to write types explicitly. TypeScript can *infer* them from context. For example:

```ts
const name = "Alice"
```

TypeScript infers `name` as `string` because you assigned a string literal. You don't need to write `const name: string = "Alice"`.

For function return types, inference often works too. If you write:

```ts
function double(n: number) {
  return n * 2
}
```

TypeScript infers the return type as `number` because `n * 2` is a number. You could add `: number` explicitly, but it's redundant.

In React components, inference is common. When you destructure props like `{ variant = "primary", size = "md" }`, TypeScript infers the types from `ButtonProps`. You don't annotate each variable.

Inference has limits. When a function returns different types based on input, or when the type is ambiguous, you may need to add annotations. And for public APIs—like component props—explicit types are preferred because they document the contract. Counterfoil uses explicit prop types for that reason.

From `src/App.tsx`:

```tsx
const [activeTab, setActiveTab] = React.useState<
  "typography" | "buttons" | "form" | "table"
>("typography")
```

Here, the type is explicit. `useState` can hold any type; without the annotation, TypeScript would infer `string` from the initial value. But the app only uses those four tab ids. By annotating `"typography" | "buttons" | "form" | "table"`, we restrict `activeTab` to valid values. If you later write `setActiveTab("invalid")`, TypeScript will error. The annotation makes the state's contract explicit.

## Why Counterfoil Uses TypeScript: Contracts, Tooling, Safety

Counterfoil is built with TypeScript for three reasons: contracts, tooling, and safety.

**Contracts.** Each component's props are a contract. The types document what the component accepts. When you use `<Button variant="primary" />`, the type system ensures `variant` is valid. When the design system adds a new variant, the type alias is updated and the compiler ensures every class map and switch handles it. Types are the single source of truth for the component API.

**Tooling.** With types, your editor can offer autocomplete. When you type `<Button v`, you get a dropdown of props including `variant`. When you type `variant="`, you get the valid values. You don't need to open the source or a doc site—the types drive the IDE. Jump-to-definition takes you to the type; hover shows the full definition. For someone learning the codebase, that's invaluable.

**Safety.** Refactoring is safer. Rename a prop, and the compiler finds every usage. Change a variant name, and you get errors wherever it's used. Add a required prop, and every caller must update. The type checker catches breaking changes before they reach production. For a design system that evolves over time, that safety reduces the cost of change.

When you read Counterfoil code, the types are not noise—they're the first place to look to understand what a component does and how to use it. (See [2.5 The Anatomy of a Component](2.5_The_Anatomy_of_a_Component_Breaking_Down_the_Code.md) for a full walkthrough of component structure.)
