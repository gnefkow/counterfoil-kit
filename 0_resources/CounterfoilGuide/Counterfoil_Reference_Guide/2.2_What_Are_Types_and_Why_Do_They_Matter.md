# 2.2: What Are Types and Why Do They Matter?

## Goal

Explain what "types" are in programming, why they matter, and what problems they solve. This chapter builds the conceptual foundation before we dive into TypeScript syntax in the next chapter. (See [2.3 Understanding TypeScript - Types in Practice](2.3_Understanding_TypeScript_Types_in_Practice.md).)

## The Problem Types Solve: Catching Mistakes Early

When you write code, you are constantly passing data between functions, components, and modules. A function expects a string. A component expects an object with specific properties. An API returns a list of items. In plain JavaScript, the language does not enforce any of these expectations. It will happily let you pass a number where a string was expected, or call a method on `undefined`, or misspell a property name. The code will run until it hits a line that fails—and then it fails in the user's browser, often in a way that's hard to trace.

Types solve this by making expectations explicit and checkable *before* the code runs. A type is a description of what a value can be: a string, a number, a list of user objects, a function that takes a click event and returns nothing. When you annotate your code with types, a tool (like TypeScript) can verify that you're using values correctly everywhere. If you pass the wrong shape of data, or call something that doesn't exist, the tool reports an error immediately—in your editor, at build time—instead of at runtime when a user clicks a button.

## Real-World Analogy: Types Are Like Labels on Boxes

Imagine a warehouse where boxes are unlabeled. You know that somewhere there's a box of "button variants" and somewhere else a box of "customer records." Without labels, you pick up a box, open it, and hope. Maybe you get the right thing. Maybe you get something else, and you only discover the mistake when you try to use it—at which point you have to trace back through the whole system to find where the wrong box came from.

Now imagine the same warehouse with labels on every box: "Button Variants (primary | secondary | tertiary)", "Customer Records (id, name, email)". When you need a button variant, you look for that label. When someone hands you a box labeled "Customer Records" but it actually contains button variants, the mismatch is obvious. You catch the error at the handoff, not later when you're trying to send an email to a variant name.

Types are those labels. They describe what's inside a variable, a function parameter, or a return value. They don't change what the code does at runtime—they're checked and then stripped away when the code is compiled to JavaScript. But while you're writing and building, they act as a contract: "this value is a string," "this object has these properties," "this function returns a number." When the contract is violated, the type checker tells you before you ship.

## What Happens Without Types

In JavaScript, you can write code like this:

```ts
function formatPrice(amount) {
  return "$" + amount.toFixed(2)
}

formatPrice(42.5)       // "$42.50" — works
formatPrice("hello")    // runtime error: toFixed is not a function
formatPrice(null)       // runtime error: cannot read property 'toFixed' of null
formatPrice()           // runtime error: cannot read property 'toFixed' of undefined
```

The function assumes `amount` is a number. Nothing enforces that. If you pass a string, `null`, or forget to pass anything, the code runs until it hits `amount.toFixed(2)` and then throws. The error might surface in a user's console, in a production logs dashboard, or in a confusing stack trace. You have to run the code, hit the failing path, and debug from there.

Similarly, with objects:

```ts
const user = { name: "Alice", age: 30 }
console.log(user.naem)  // undefined — typo, no error
```

You meant `user.name` but typed `user.naem`. JavaScript doesn't complain. You get `undefined`, and if that value flows into a calculation or a UI, you get odd behavior or another runtime error downstream. Finding the source of `undefined` in a large codebase can be tedious.

In a component-based UI like React, the problem scales. A `Button` component might expect props like `variant`, `size`, and `onClick`. In plain JavaScript, nothing stops a developer from passing `variant="prmary"` (typo), `size={42}` (wrong type), or `onClick="doSomething"` (string instead of function). The component might render with wrong styles, or crash when clicked, or silently do nothing. The mistake is only discovered when someone runs the app and uses that button.

## What Happens With Types

With types, the same mistakes are caught as soon as you write or save the file. Using TypeScript, the `formatPrice` example becomes:

```ts
function formatPrice(amount: number) {
  return "$" + amount.toFixed(2)
}

formatPrice(42.5)       // OK
formatPrice("hello")    // error: Argument of type 'string' is not assignable to parameter of type 'number'
formatPrice(null)       // error: Argument of type 'null' is not assignable to parameter of type 'number'
formatPrice()           // error: Expected 1 arguments, but got 0
```

The type checker sees the mismatches and reports them. You fix the call sites or the function before the code ever runs.

For the typo:

```ts
const user: { name: string; age: number } = { name: "Alice", age: 30 }
console.log(user.naem)  // error: Property 'naem' does not exist on type '{ name: string; age: number }'
```

The editor and build will flag the typo immediately.

In a component library like Counterfoil, types define the *public API* of each component. For example, the `Button` component (from `src/components/primitives/Button.tsx`) declares that `variant` must be one of `"primary" | "secondary" | "tertiary"` and so on. If you write:

```tsx
<Button variant="prmary">Save</Button>
```

TypeScript reports that `"prmary"` is not a valid value for `variant`. You get autocomplete suggesting the valid options. You don't need to open the Button source to remember the exact string—the type system is the documentation.

## Types and Object-Oriented Programming

If you're familiar with object-oriented programming, you've already seen a form of types: classes. A class defines the shape of its instances—what properties and methods they have. When you call `user.getName()`, the compiler (in languages like Java or C#) checks that `user` is an object that has a `getName` method. That's type checking.

TypeScript and similar type systems extend this idea beyond classes. They let you describe the shape of *any* value: plain objects, arrays, functions, strings, numbers. You don't need a class to have a type. An object literal has a type based on its structure. A function has a type based on its parameters and return value. This is often called *structural typing*: if two objects have the same shape (same properties with compatible types), they're considered compatible, even if they weren't created from the same class.

In Counterfoil, components use object types and union types more than classes. For example, `ButtonProps` describes an object with optional `variant`, `size`, `children`, and so on. The `Button` function receives that object. TypeScript ensures that every place you use `<Button ... />`, you pass an object that matches `ButtonProps`. No class required—just a description of the shape.

## Clarity and Self-Documentation

Types also serve as built-in documentation. When you open a component file and see:

```ts
type ButtonProps = {
  variant?: ButtonVariant
  size?: ButtonSize
  disabled?: boolean
  onClick?: () => void
  children?: React.ReactNode
}
```

you can read, without running anything, what the component accepts. `variant` is optional and must be one of the `ButtonVariant` values. `onClick` is an optional function that takes no arguments and returns nothing. `children` is optional React content. You don't need a separate doc file or to memorize the API—the types are the contract.

This is especially valuable when working with others. A new developer joining the team can explore the codebase by reading type definitions. The IDE can show them the types on hover. They get immediate feedback when they pass the wrong props. The types reduce the need for tribal knowledge and long onboarding docs.

## Build-Time Safety, Not Runtime Magic

It's important to understand: types exist only at development and build time. TypeScript compiles to JavaScript. The compiled output has no type annotations—they're erased. At runtime, you're still running plain JavaScript. Types don't make your code faster or add behavior; they help you write correct code before it runs.

That means type checking cannot catch every possible bug. It can't verify that your API actually returns the shape you declared. It can't prevent logic errors (e.g., using the wrong variable in a calculation). It can't replace tests. But it catches a large class of mistakes—wrong types, missing properties, typos, wrong number of arguments—that would otherwise surface as runtime errors. For a design system or component library, that's invaluable: you want consumers to get clear errors when they use the API incorrectly, not mysterious failures in production.

## Why Counterfoil Uses Types

Counterfoil is built with TypeScript. Every component has explicit prop types. Variants, sizes, and other options are defined as union types (e.g., `"primary" | "secondary"`). When you use a Counterfoil component, your editor and build tool verify that you're passing valid props. If you add a new variant to the design system, you add it to the type first; the type checker then ensures that every place that handles variants is updated. Types are central to how the toolkit stays coherent as it evolves.
