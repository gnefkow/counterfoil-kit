# 2.4: Understanding React - Components, Props, and JSX

## Goal

Explain what React is, how components work, what JSX is, and how props pass data between components. By the end of this chapter, you'll be able to read a Counterfoil component file and understand its overall shape: imports, types, function, and return statement. This chapter is the runtime counterpart to the type-level concepts covered in (See 2.3 Understanding TypeScript - Types in Practice.)

## What React Is: A Library for Building UIs

React is a JavaScript library for building user interfaces. It was created at Facebook (now Meta) and is the most widely used front-end library in the industry. React is not a framework—it doesn't dictate how you structure your application, fetch data, or handle routing. It does one thing well: it lets you describe what the UI should look like based on data, and it efficiently updates the browser when that data changes.

The core idea is declarative rendering. Instead of writing step-by-step instructions to manipulate the DOM ("find this element, change its text, add a class"), you describe the desired result ("render a button with this label and these styles") and React figures out how to make the browser match. When data changes, you describe the new result and React updates only the parts that changed.

Counterfoil is built on React. Every component in the toolkit—`Button`, `Stack`, `Text`, `InputField`—is a React component. Understanding React's component model is essential to reading the codebase.

## Components: Reusable Pieces of UI

A **component** is a reusable, self-contained piece of user interface. Think of it like a LEGO brick: each brick has a defined shape and purpose, and you snap them together to build something larger. A `Button` component renders a button. A `Stack` component arranges children vertically with consistent spacing. An `InputField` component renders a label, an input control, and optional help text—all wired together.

In React, a component is a function. It takes input (called **props**), does some work, and returns a description of what should appear on screen. That description is written in **JSX**, which we'll cover next.

Here's the simplest possible component:

```tsx
function Greeting() {
  return <p>Hello, world</p>
}
```

`Greeting` is a function. It takes no input and returns a paragraph element. That's it—that's a component.

In Counterfoil, components are more involved, but the shape is the same. From `src/components/primitives/Stack.tsx`:

```tsx
export function Stack({ children, gap = "m", className }: StackProps) {
  return (
    <div className={[`flex flex-col ${gapClasses[gap]}`, className].filter(Boolean).join(" ")}>
      {children}
    </div>
  )
}
```

`Stack` is a function. It receives props (`children`, `gap`, `className`), computes a class string, and returns a `<div>` with that class string and the children inside it. The function is the component; the return value is the UI.

Components are **composable**. You build larger UIs by combining smaller components. In Counterfoil's `InputField`, the component uses `Stack`, `Inline`, `Text`, and `Input`:

```tsx
export function InputField({ label, description, helpText, errorText, /* ... */ }: InputFieldProps) {
  return (
    <Stack gap="xs">
      <Inline gap="xs" align="center">
        <Text size="body1" hierarchy="primary">{label}</Text>
      </Inline>
      <Input type={type} value={value} placeholder={placeholder} onChange={onChange} />
      {helpText && !hasError ? (
        <Text size="body2" hierarchy="tertiary">{helpText}</Text>
      ) : null}
    </Stack>
  )
}
```

Each of those inner components—`Stack`, `Inline`, `Text`, `Input`—is itself a function that returns UI. React renders the tree by calling each function, getting its return value, and recursively rendering any components found inside. The result is a tree of HTML elements in the browser.

This composability is why React aligns with design-system thinking. Designers compose screens from components; React developers do the same thing in code. The mental model transfers directly.

## JSX: HTML-Like Syntax That's Actually JavaScript

JSX is the syntax React uses to describe UI. It looks like HTML:

```tsx
<Button variant="primary" size="md">Save</Button>
```

But it's not HTML. It's JavaScript with a special syntax extension. When the code is compiled (by tools like Vite and TypeScript), JSX is converted to regular function calls. The line above becomes something like:

```ts
React.createElement(Button, { variant: "primary", size: "md" }, "Save")
```

You don't need to think about `React.createElement`—you write JSX and the build tool handles the conversion. But knowing this helps demystify the syntax: every `<Component>` tag is a function call, and every attribute is a property in an object passed to that function.

### How to Read JSX

**Tags** correspond to either HTML elements or React components. Lowercase tags are HTML elements; capitalized tags are components.

- `<div>` — HTML `div` element
- `<button>` — HTML `button` element
- `<Button>` — the `Button` component (a function you or someone else wrote)
- `<Stack>` — the `Stack` component

**Attributes** become properties on the props object. In HTML, you write `<input type="text" />`. In JSX, the syntax is the same, but attributes can accept any JavaScript value using curly braces:

```tsx
<Button variant="primary" disabled={true} onClick={() => handleSave()}>
  Save
</Button>
```

- `variant="primary"` — a string literal, passed as `props.variant`
- `disabled={true}` — a boolean, passed as `props.disabled`
- `onClick={() => handleSave()}` — a function, passed as `props.onClick`

Curly braces `{}` in JSX mean "evaluate this JavaScript expression." A string in quotes doesn't need them; everything else does.

**Children** are whatever appears between the opening and closing tags. In `<Button>Save</Button>`, `"Save"` is the child. Children are passed as `props.children`. They can be text, other components, or JavaScript expressions:

```tsx
<Stack gap="lg">
  <Text size="h1">Dashboard</Text>
  <Button variant="primary">Create New</Button>
</Stack>
```

Here, `Stack` receives two children: a `Text` component and a `Button` component. `Stack` renders them inside its `<div>`, arranged vertically with the specified gap.

**Self-closing tags** are used when a component has no children: `<Input />` is equivalent to `<Input></Input>`. You'll see this often for components like `Input`, which renders an `<input>` element and doesn't wrap other content.

### Expressions in JSX

Inside curly braces, you can use any JavaScript expression. This is how components render dynamic content:

```tsx
<Text size="body1" hierarchy={disabled ? "tertiary" : "primary"}>
  {label}
</Text>
```

This is from `InputField`. The `hierarchy` prop is determined by a ternary expression: if `disabled` is true, use `"tertiary"`; otherwise use `"primary"`. The child `{label}` inserts whatever string was passed as the `label` prop. Both are evaluated as JavaScript at render time.

You can also conditionally render entire elements:

```tsx
{description ? (
  <Text size="body2" hierarchy="secondary">
    {description}
  </Text>
) : null}
```

Read this as: "If `description` is truthy (not `undefined`, not empty), render a `Text` component with that description. Otherwise, render nothing (`null`)." This pattern is everywhere in Counterfoil. Components conditionally show help text, error messages, icons, and other optional pieces based on the props they receive.

## Props: Passing Data to Components

**Props** (short for "properties") are how you pass data to a component. They're the component's input—like function arguments, but for UI.

When you write:

```tsx
<Button variant="primary" size="lg" onClick={() => alert("clicked")}>
  Save
</Button>
```

you're passing three props to `Button`: `variant`, `size`, and `onClick`, plus `children` (the text `"Save"`). Inside the `Button` function, those values are available as properties of the props object.

Props flow **one direction: parent to child**. A parent component decides what data to pass; the child component receives it and renders accordingly. The child cannot modify its own props—they're read-only. If the child needs to communicate back to the parent (e.g., "I was clicked"), the parent passes a function as a prop (like `onClick`), and the child calls that function.

This one-way data flow keeps things predictable. When you read a component, you can look at its props to understand what data it depends on and what callbacks it can trigger. The type definition is the contract. From `src/components/primitives/Button.tsx`:

```ts
type ButtonProps = {
  children?: React.ReactNode
  variant?: ButtonVariant
  size?: ButtonSize
  width?: ButtonWidth
  disabled?: boolean
  onClick?: () => void
  icon?: React.ReactNode
  iconPosition?: "left" | "right"
  leftIcon?: React.ReactNode
  rightIcon?: React.ReactNode
  "aria-label"?: string
}
```

Every prop is optional (all have `?`). The types tell you what values are valid. `variant` must be a `ButtonVariant` string; `onClick` must be a function with no arguments and no return value; `children` can be any renderable React content. If you pass something that doesn't match, TypeScript will error. (See 2.3 Understanding TypeScript - Types in Practice.)

### Props with Default Values

When a prop is optional, the component often specifies a default. You see this in the function signature using JavaScript's destructuring defaults:

```tsx
export function Button({
  variant = "primary",
  size = "md",
  width = "hug",
  disabled = false,
  // ...
}: ButtonProps) {
```

If you render `<Button>Save</Button>` without specifying `variant`, `size`, or `width`, the component uses `"primary"`, `"md"`, and `"hug"` respectively. You only need to pass a prop when you want something other than the default.

Defaults make the component easy to use out of the box while still allowing full customization. This is a deliberate design pattern in Counterfoil: the simplest usage should require the fewest props.

## Reading Component Code Structure

Every Counterfoil component follows a consistent structure. Once you recognize the pattern, you can read any component file quickly. There are four sections to look for:

### 1. Imports: What the Component Needs

The top of the file lists `import` statements. These declare what the component depends on: React itself, other components, types, icons, utilities.

From `src/components/form/InputField.tsx`:

```tsx
import React from "react"

import { Input } from "./Input"
import { Inline, Stack, Text } from "../primitives"
```

Read this as: "InputField needs React (for JSX), the `Input` component from the same folder, and `Inline`, `Stack`, and `Text` from the primitives folder." The imports tell you, at a glance, what building blocks this component uses. If you see an unfamiliar component in the JSX, check the imports to find where it lives.

Compare with `src/components/primitives/Button.tsx`:

```tsx
import React, { useEffect, useRef } from "react"
```

Button only imports from React. It doesn't depend on other Counterfoil components—it's a primitive, a building block that others build on. The imports reflect that: simpler components have fewer imports; composite components import the things they assemble.

### 2. Type Definitions: The Component's Public API

After imports, you'll find type definitions. These describe the props the component accepts—its public interface. In Counterfoil, this section is labeled with a comment:

```tsx
/* ======================================================
   Public API
   ====================================================== */

type StackGap = "xs" | "sm" | "m" | "lg" | "xl"

type StackProps = {
  children: React.ReactNode
  gap?: StackGap
  className?: string
}
```

This is from `Stack`. The types tell you everything about how to use the component: `children` is required (the content to stack), `gap` is optional and must be one of five values, and `className` is an optional string for additional styling. You don't need to read the component function to know the API—the types are the specification.

For details on reading these type definitions, (See 2.3 Understanding TypeScript - Types in Practice.)

### 3. Component Function: The Implementation

The component function is where the work happens. It receives props, performs any necessary logic, and returns JSX. Here is `Stack` in full:

```tsx
export function Stack({ children, gap = "m", className }: StackProps) {
  return (
    <div className={[`flex flex-col ${gapClasses[gap]}`, className].filter(Boolean).join(" ")}>
      {children}
    </div>
  )
}
```

The function signature tells you this is an exported function (available to other files) called `Stack`. The props are destructured directly in the parameter list: `{ children, gap = "m", className }` pulls out those three properties, with a default of `"m"` for `gap`. The `: StackProps` annotation tells TypeScript to check incoming props against the `StackProps` type.

### 4. Return Statement: The JSX That Gets Rendered

The `return` statement contains the JSX—the description of what should appear on screen. In `Stack`, it returns a `<div>` with computed classes and the children inside. In `Button`, the return is more complex:

```tsx
return (
  <button
    ref={buttonRef}
    type="button"
    disabled={disabled}
    onClick={onClick}
    className={finalClassName}
    aria-label={ariaLabel}
  >
    {leftIconElement}
    {children}
    {rightIconElement}
  </button>
)
```

Read the return statement as a description of the rendered output. This button element has a ref, a type, disabled and onClick attributes, a className computed from the variant/size/width props, and an aria-label. Inside the button, it renders an optional left icon, the children (text), and an optional right icon. Each `{}` expression is evaluated at render time.

### Putting It Together

When you open a Counterfoil component file, read it in this order:

1. **Imports** — what does this component depend on?
2. **Types** — what props does it accept? What's required vs. optional?
3. **Component function** — scan the logic between the function signature and the `return`
4. **Return statement** — what HTML/JSX does it produce?

This top-to-bottom reading order matches how the code is organized. (See 2.5 The Anatomy of a Component - Breaking Down the Code for a full, line-by-line walkthrough of a real component.)

## State Basics: `useState`

So far, everything has been about **props**: data passed into a component from its parent. But sometimes a component needs to track data that changes over time on its own—data that the component itself manages. This is called **state**.

**State is data that can change.** When state changes, React re-renders the component with the new data. The UI updates automatically.

React provides a built-in function called `useState` for this. Here's a real example from `src/App.tsx`:

```tsx
const [activeTab, setActiveTab] = React.useState<
  "typography" | "buttons" | "form" | "table"
>("typography")
```

Break this down:

- `React.useState(...)` creates a piece of state.
- The initial value is `"typography"`—the tab that's selected when the page first loads.
- `useState` returns an array with two items:
  - `activeTab` — the current value of the state
  - `setActiveTab` — a function to update the state
- The type annotation `<"typography" | "buttons" | "form" | "table">` restricts the state to those four string values.

When the user clicks a different tab, the code calls `setActiveTab("buttons")` (or whichever tab was clicked). React re-renders the `App` component with `activeTab` now equal to `"buttons"`, and the UI updates to show the buttons panel instead of the typography panel.

The pattern is always the same:

```tsx
const [currentValue, setCurrentValue] = React.useState(initialValue)
```

You read it as: "Create a state variable called `currentValue`, starting at `initialValue`. Call `setCurrentValue` to change it."

### When Components Need State

Not every component needs state. In fact, most Counterfoil components are **stateless**—they receive props and render UI, with no internal data to track. `Button`, `Stack`, `Text`, `Inline`—none of these manage state. They render whatever props they're given.

State appears when a component needs to **remember something between renders**. Common cases:

- **Which tab is selected** — the `App` component tracks `activeTab` so it knows which panel to show.
- **What the user typed** — a form component might track the input value so it can display it and respond to changes.
- **Whether a dropdown is open** — a menu component tracks open/closed so it can show or hide the options.

In Counterfoil, you'll encounter state primarily in application code (like `App.tsx`) and in interactive components that manage their own behavior. The primitive components in the toolkit are mostly stateless by design—they receive data via props and let the parent manage state. This separation keeps the primitives simple and reusable.
